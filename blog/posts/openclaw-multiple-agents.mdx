本博客，我将尝试在 OpenClaw 接入多款 AI 大模型，对接到多个 AI 代理，让大家各司其职，在自己擅长的领域里发光发热，为我打工！

## 对接多款 AI 大模型

首先，认清自己的需求，我并不需要多么强大的模型来跑什么复杂任务，因此选择排行榜上佼佼的顶级模型是极不划算的，每天定时任务抓取一次网页就烧几块钱这谁顶得住？其次，直接使用模型官方 API 往往意味着更高的成本，而选用三方 API 套餐往往更加划算，区别在于后者存在速率限制和次数限制，也爆出新闻提到某些转发商混用模型降低自己成本，但是我是非深度用户，性价比就是我第一顺位要考虑的事情，小小的不愉快总能用其他方式解决。

综上所述，尽管国外的几款大模型仍领先国产大模型一个身位，但我并没有追求使用它们。这里，我以官方 API 的形式对接了几款国内的大模型，配合以前订阅的 Copilot，满足自己的日常使用需要：

### 千问

阿里云通义千问的大模型服务平台目前命名为[百炼](https://modelstudio.console.alibabacloud.com/)，在文档的 “Coding Plan” - “接入 AI 工具” - [“OpenClaw” 章节](https://modelstudio.console.alibabacloud.com/ap-southeast-1/?tab=doc#/doc/?type=model&url=3023085)中提供了接入 OpenClaw 的完整教程。

接入完成后，可以用 `openclaw models list --provider <PROVIDER_NAME>` 来查看指定提供者的模型列表，以及身份验证是否成功：

```bash
$ openclaw models list --provider bailian

🦞 OpenClaw 2026.3.7 (42a1394) — I'm like tmux: confusing at first, then suddenly you can't live without me.

Model                                      Input      Ctx      Local Auth  Tags
bailian/qwen3.5-plus                       text+image 977k     no    yes   configured,alias:Qwen 3.5 Plus
bailian/qwen3-coder-next                   text       256k     no    yes   configured,alias:Qwen 3.5 Coder Next
```

通过百炼接入的 AI 模型是付费 API 的形式，当新人赠送额度用完后，就需要储值使用了。

另外，OpenClaw 官方提供了 `qwen-portal-auth` 插件用于接入 Qwen Coder 和 Qwen Vision 模型，目前这两款模型有每天免费的使用额度，可以当羊毛薅来用一用：

```bash
# 启用插件
openclaw plugins enable qwen-portal-auth
# 认证登录
openclaw models auth login --provider qwen-portal
```

### Kimi

据社区里的用户推荐，月之暗面的 Kimi 模型有着较好的任务规划能力，适合拿来做 OpenClaw 多智能体聚落的“主脑”。当然，国内的大模型一般也会有比较好的本土化调教，理论上对中文输入的理解能力更强。

首先注册登录[月之暗面控制台](https://platform.moonshot.cn/console/account)账号，创建好项目和 API Key 后，参考 OpenClaw 官方文档提供的[月之暗面模型接入教程](https://docs.openclaw.ai/providers/moonshot)，完成接入配置。

验证是否接入成功：

### Copilot

我从去年开始订阅 Github Copilot 作为代码编程的 AI 助手，10 美元每月的价格还要啥自行车。目前，它支持免费使用 GPT 5 mini 与仍服役的几款老 GPT 模型，也支持使用每月的额度调用 Claude Opus 4.6 等强力模型。

在 Github 官网完成 Copilot 订阅后，按照 OpenClaw 官方文档提供的 [Github Copilot 接入教程](https://docs.openclaw.ai/providers/github-copilot)，完成接入配置。

特别的，需要手动修改 OpenClaw 的配置文件，添加想要使用的模型列表。例如，笔者配置的模型列表如下：

```json
// ~/.openclaw/config.json
{
  "agents": {
    "defaults": {
      "models": {
        "github-copilot/claude-opus-4.6": {
          "alias": "Copilot Claude Opus 4.6"
        },
        "github-copilot/claude-sonnet-4.6": {
          "alias": "Copilot Claude Sonnet 4.6"
        },
        "github-copilot/gemini-3.1-pro-preview": {
          "alias": "Copilot Gemini 3.1 Pro (Preview)"
        },
        "github-copilot/gpt-5.4": {
          "alias": "Copilot GPT 5.4"
        },
        "github-copilot/gpt-5.3-codex": {
          "alias": "Copilot GPT 5.3 Codex"
        },
        "github-copilot/gpt-5-mini": {
          "alias": "Copilot GPT 5 Mini"
        },
        "github-copilot/gpt-4.1": {
          "alias": "Copilot GPT 4.1"
        }
      }
    }
  }
}
```

现在，除了 GPT 5.4 模型暂时无法使用外（在之前的 OpenClaw 2026.3.2 版本，Claude Opus 4.6 和 GPT 5.3 Codex 等新模型都不可用，撰写此文时插件已更新支持了），其它配置好的模型都能正常调用了。

超出每月限额后，再调用高级模型会按 0.04 美元每个请求的价格收费，请确保自己的[预算配置](https://github.com/settings/billing/budgets)合理，避免烧走不必要的资金。

对应执行命令：

```bash
openclaw agents add daily --workspace ~/.openclaw/workspace-daily --model github-copilot/gpt-4.1
openclaw agents add daily-cn --workspace ~/.openclaw/workspace-daily-cn --model bailian/qwen3.5-plus
openclaw agents add coder --workspace ~/.openclaw/workspace-coder --model bailian/qwen3-coder-next
openclaw agents add coder-pro --workspace ~/.openclaw/workspace-coder-pro --model github-copilot/gpt-5.3-codex
openclaw agents add creativity --workspace ~/.openclaw/workspace-creativity --model github-copilot/gemini-3.1-pro-preview
openclaw agents add expert --workspace ~/.openclaw/workspace-expert --model github-copilot/claude-opus-4.6
```

赋予不同代理合适的权限，例如（省略其它配置）：

```json
{
  "tools": {
    "profile": "messaging"
  },
  "agents": {
    "list": [
      {
        "id": "main",
        "tools": {
          "profile": "full"
        },
        "subagents": {
          "allowAgents": ["*"]
        }
      },
      {
        "id": "daily"
      },
      {
        "id": "daily-cn"
      },
      {
        "id": "coder",
        "tools": {
          "profile": "coding"
        }
      },
      {
        "id": "coder-pro",
        "tools": {
          "profile": "coding"
        }
      },
      {
        "id": "creativity",
        "tools": {
          "alsoAllow": [
            "read",
            "write",
            "edit",
            "web_search",
            "web_fetch",
            "memory_search",
            "memory_get",
            "image",
            "tts"
          ]
        }
      },
      {
        "id": "expert",
        "tools": {
          "alsoAllow": ["web_search", "web_fetch"]
        }
      }
    ]
  }
}
```

上面的配置表示，代理默认使用 `messaging` 工具权限，`main` 代理拥有 `full` 全部工具权限，`coder` 和 `coder-pro` 代理拥有 `coding` 工具权限，`creativity` 和 `expert` 代理额外允许指定工具的权限。特别的，这里为 `main` 代理设置了 `subagents.allowAgents = ["*"]`，使得其能发现并调用别的代理为它工作。

## 开始工作

完成以上配置后，让我们来检验一下多代理协同工作的效果吧！

### `git rebase` 的用法

首先，我向 `main` 代理发出任务：「找到最合适的代理，询问它“git rebase 应该怎么使用？”」。

它检索了已有代理列表，找到了合适执行此任务的代理，回复道：「已派发给 **coder-pro** 代理来回答关于 git rebase 的问题，请稍等片刻～ ✨」。

查看 `coder-pro` 代理接收到的消息，内容如下：

```plaintext
[Subagent Context] You are running as a subagent (depth 1/1). Results auto-announce to your requester; do not busy-poll for status.

[Subagent Task]: git rebase 应该怎么使用？请详细解释 git rebase 的工作原理、常用场景、基本用法以及注意事项。
```

`coder-pro` 代理理解了问题，给出了详细回答：

```md

---

- merge：把两条分支“汇合”，通常会生成一个 merge commit，历史保留分叉形状。
- rebase：把你当前分支上的提交“摘下来”，然后在新基点上重新应用，历史更线性。

## 2) 工作原理（重点）
```
